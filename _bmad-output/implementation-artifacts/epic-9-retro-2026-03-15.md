## Challenges & Learnings

### 1. Vertical Slice Gap — Backend-First Story Design
- **Challenge:** Every originally planned story (9.1-9.4) was backend-only. Dashboard visibility deferred. This created a wave of 8 catch-up dashboard stories (9-7 through 9-14, 9-21) and delayed operator validation of the features being built.
- **Root cause:** Stories structured as horizontal layers (backend → API → dashboard) rather than vertical slices (feature visible end-to-end).
- **Impact:** Arbi couldn't verify correlation cluster assignments until 9-7 made them visible. Couldn't confirm capital efficiency gating until 9-9 added APR column. Story 9-9 wasn't a UX enhancement — it was the first time anyone could observe detection pipeline calculations against reality.
- **Key insight (Arbi):** "An unobservable feature is an unvalidated feature, regardless of how many unit tests it passes."
- **Resolution:** Team agreement #18: vertical slice minimum — every story with a backend capability includes at least minimal dashboard observability.

### 2. Internal Subsystem Verification Philosophy
- **Challenge:** Applied mock fidelity lesson (from Epic 8) outward toward platform APIs (Zod schemas) but not inward toward own subsystems. WebSocket code connected but never subscribed — undetected for 9 epics. Health monitoring showed 22-35% degradation time and it was accepted as normal.
- **Root cause:** Tests validated that receiving code handles data correctly, not that data actually arrives through claimed channels.
- **Examples:** 9-20 (WebSocket subscriptions empty), 9-15 (epoch-zero false degradation), 9-17 (stale APR preserved across cycles), 9-18 (TP formula zero-floor with specific input ranges)
- **Key insight (Dana):** "We applied the mock fidelity lesson from Epic 8 outward but not inward. Zod schemas catch Kalshi sending unexpected data. Nothing catches us silently not requesting data we claim to depend on."
- **Resolution:** Team agreement #19: internal subsystem verification mandate — tests must verify data actually arrives, not just that receiving code handles it correctly.

### 3. Correctness Gaps Living Undetected
- **Challenge:** 9-20 (WebSocket), 9-15 (epoch-zero health), 9-17 (stale APR), 9-18 (TP formula flooring) — these aren't planning failures or scope additions. They're things that were supposed to work and didn't. The test suite validated internal consistency but not end-to-end operational correctness.
- **Root cause (Winston):** Tests validate internal logic but not operational behavior. Mocks for health monitoring never asked "is WebSocket data actually flowing." Exit threshold tests never exercised the formula with high-spread/low-edge inputs.
- **Key insight (Arbi):** "These are validation philosophy issues. No amount of vertical slicing fixes 'WebSocket code that connects but never subscribes' if your test suite never asks 'is data actually arriving through this channel?'"
- **Resolution:** Combined with internal verification mandate + investigation-first pattern for ambiguous root causes.

### 4. Tech Debt Compound Interest Pattern
- **Challenge:** Three consecutive epics (6.5, 8, 9) generated and carried internal debt. Epic 9 paid down Epic 8's debt (branded types, Zod schemas, calibration persistence) while simultaneously creating its own (59-finding UX audit in 9-21, paper/live boundary gaps, WebSocket subscription gap). Each layer of the epic created cleanup work for the next layer.
- **Key insight (Arbi):** "Every epic will generate internal debt. The question is whether we pay it within the epic or carry it forward. Epic 9 paid it within the epic — that's why we had 20 course corrections instead of 6 stories plus a backlog of 'fix later' items. That's the right choice, but it means epic sizing needs to account for it."
- **Resolution:** Team agreement #22: budget 30-40% beyond planned stories for internal corrections.

### 5. Paper Trading Boundary is First-Class
- **Challenge:** Story 9-16 paper/live risk state isolation hit post-deploy bug — paper positions tried to verify against real platform APIs (simulated fills don't exist). Paper mode has fundamentally different execution semantics, not just a boolean flag.
- **Key insight (Dana):** "Paper mode is a different execution context, not just a flag. We need tests that specifically exercise the boundary between paper and live."
- **Critical for Epic 10:** Story 10.3 (automatic single-leg management) behavior must differ in paper vs live. If validated only in paper mode, live transition discoveries could be costly.
- **Resolution:** Team agreement #20: paper/live boundary is first-class. Dedicated test suite for `isPaper` branches.

### 6. Recurring MEDIUM Code Review Categories
- **Challenge:** Same categories appeared across 10+ stories: (1) incomplete test assertions (10 stories), (2) dead code / duplicate utilities (4 stories), (3) missing DTO fields / validation at API boundaries (4 stories). Caught every time by review, never prevented structurally.
- **Root cause analysis needed (Arbi):** Categorize MEDIUMs by *why the dev agent produced them*. "Dev agent doesn't know to add ValidationPipe" (CLAUDE.md fix) is different from "dev agent writes `toBeDefined()` instead of specific assertions" (test template or linter rule). Solution depends on cause.
- **Resolution:** Action item #3: Charlie + Dana analyze actual review records, propose structural guards.

### 7. Nonlinear Velocity (Reframed as Insight)
- First half (9-0-1 through 9-8): slow — fighting accumulated debt (branded types touching 27 files, Zod requiring build-break-then-fix, Kalshi API migration, matches page full redesign)
- Second half (9-9 through 9-21): dramatically faster — composing on shared abstractions (DataTable, useUrlTableState, shared pure functions, InfoTooltip)
- **Key insight (Arbi):** "Foundational investment has a J-curve. The first half was slower because it was building infrastructure. The second half was faster because it was consuming that infrastructure. For Epic 10, this means we should expect WebSocket subscription and foundational continuous monitoring patterns to feel slow. That's an investment, not a velocity problem."
- **Resolution:** Team agreement #21: plan for the J-curve, don't fight it.

## Key Insights

1. **An unobservable feature is an unvalidated feature** — the dashboard is a validation surface, not just a display layer. Building it last means validating last.
2. **Mock fidelity applies inward, not just outward** — internal subsystem verification needs the same rigor as platform API validation. Nothing in our test infrastructure caught us silently not requesting data we claim to depend on.
3. **Catching a bug 15 times in review works; preventing it once is better** — recurring MEDIUM categories need structural prevention, not just reliable detection.
4. **Foundational investment has a J-curve** — early-epic infrastructure stories feel slow because they're building the foundation. Late-epic stories move fast because they compose on it. Plan for the curve.
5. **Story-or-drop discipline is load-bearing for project maintainability** — 20 corrections formalized means every decision is auditable and every pattern is learnable. Nine months from now, the answer to "why is primary_leg populated for matches after March 2026?" is in story 9-8's record.
6. **Every epic generates internal debt; the question is when you pay it** — paying within the epic (20 course corrections) is better than carrying forward, but requires budgeting 30-40% capacity for corrections.
7. **Epic 10 raises the consequence bar** — moving from "detect and report" to "detect and act autonomously." The cost of a correctness gap changes from "wrong data on screen" to "wrong trade executed." Every process improvement committed today exists because Epic 10 demands it.
