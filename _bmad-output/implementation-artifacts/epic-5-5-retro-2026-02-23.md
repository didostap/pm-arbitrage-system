## Challenges & Learnings

### 1. Parameter Sprawl on the Hot Path — Correctness Risk
- **Challenge:** `handleSingleLeg` grew to 16 parameters. Event constructors (especially `SingleLegExposureEvent`) have 12+ params.
- **Why it matters:** `handleSingleLeg` is on the hot path (NFR-P3: <100ms between leg submissions). Two adjacent booleans in a 16-param method — if swapped, TypeScript won't complain and tests may not catch it if both default to `false`. Silent corruption of execution state discoverable only via P&L anomaly days later.
- **Arbi's framing:** "The refactor to a context object isn't cosmetic. It's a correctness concern disguised as a style concern."
- **Resolution:** `SingleLegContext` interface to be defined in monitoring prep story. Migration is a follow-on story. Type definition creates the contract; migration has an explicit path.
- **Learning:** Parameter sprawl on hot paths is a correctness risk, not a style issue. Distinguish the two explicitly.

### 2. Story 5.5.3 Code Review Density — Complexity Drives Late-Caught Issues
- **Challenge:** 3 High-severity code review fixes in Story 5.5.3 — highest in the epic.
- **Root cause:** 5.5.3 was the most specification-complex story: `isPaper`/`mixedMode` threading across execution, exit-management, core, and common/events (4 modules). Complex specs produce late-caught issues.
- **Arbi's framing:** "The signal isn't 'the dev made more mistakes.' The signal is 'complex specs need earlier design review, not just code review at the end.'"
- **Contrast:** Stories 5.5.0-5.5.2 each touched 1-2 modules and had lower review density. 5.5.3 touched 4 modules.
- **Resolution:** LAD design review mandatory for multi-module stories (3+ modules). Timing: before implementation, during previous story's window.
- **Learning:** Design review before implementation for complex stories. Code review catches implementation bugs; design review catches specification bugs.

### 3. DI Chain Assumptions Need Verification
- **Challenge:** Story 5.5.3 spec claimed connector tokens were transitively available to `CoreModule` via nested module imports. Code review discovered NestJS doesn't work that way — `CoreModule` needed explicit `ConnectorModule` import.
- **Arbi's framing:** "This is a category of assumption that will recur as we build Epic 6 and the monitoring module starts depending on execution state."
- **Resolution:** DI resolution integration test added to DoD for any story adding cross-module injections. Boots full module graph, confirms injection.
- **Learning:** Reason about DI chains during spec writing; verify them during implementation. Integration smoke tests close the gap.

### 4. Risk Budget Contamination in Paper Mode
- **Challenge:** Paper trades consume live risk budget during a session (RiskManager unaware of paper mode, interface frozen). Cleaned at restart, but during a session, paper trades can trigger daily loss halt and block live opportunities.
- **Arbi's concern:** MVP success gate requires <3 single-leg exposure events requiring manual intervention. Paper-triggered halt during live opportunity window distorts gate data — not a safety risk, but a measurement risk.
- **Resolution:** New `PaperBudgetDriftEvent` to be defined in monitoring prep story. Monitoring module in Epic 6 routes it to Telegram as operator alert: "Paper trades consumed X% of risk budget, restart recommended." Not a halt, just visibility.
- **Architectural decision needed:** Emission location — execution service (per paper trade) vs scheduled check (periodic). Scheduled check preferred (monitoring concern, not execution concern). Decision captured in prep story spec.
- **Learning:** Known limitations are acceptable when mitigated and monitored. The key is turning a known limitation into an observable event.

### 5. Isolation-First Principle — Deliberate Discipline
- **Insight (Arbi):** Every time in Epic 5.5 where we were tempted to make `RiskManager` paper-aware or add `if (paperMode)` conditionals to `ExecutionService`, we pushed the concern to the boundary instead:
  - `isPaper` lives in repository query defaults
  - `mixedMode` lives in event payloads
  - Mode detection lives in `getHealth()` at the connector boundary
  - Alert routing lives in a new event, not a new method
- **Result:** Every module that exists today is unchanged for live trading
- **Arbi's warning:** "The temptation to 'just add a flag to MonitoringService' is going to be real in Epic 6. Resist it the same way we resisted it here."
- **Learning:** Isolation-first is a deliberate architectural choice that needs to stay deliberate. It's not instinct — it's discipline.

## Key Insights

1. **Isolation-first principle** — push mode-awareness to boundaries (repository filters, event payloads, decorator wrappers), never into module core logic. This is what made paper trading a zero-change addition for live modules.
2. **Design review timing matters** — complex multi-module stories need LAD review before implementation, not just code review after. 5.5.3's three Highs are the evidence.
3. **Compound returns on disciplined foundations** — Sprint 0's event architecture, Epic 4.5's DoD gates, Epic 5's interface freeze all paid forward into 5.5's clean delivery.
4. **Story-or-drop is permanent** — 5/5 follow-through. Sixth consecutive epic. No exceptions.
5. **Prep work embedded in epics** — Story X.0 pattern works. Monitoring prep story created before retro closes.
6. **Parameter sprawl on hot paths is correctness, not style** — distinguish explicitly and define context object types proactively.
7. **DI chains: reason in specs, verify in tests** — integration smoke tests close the gap between assumption and confirmation.
