## Successes & Strengths

1. **Design spike gate (Agreement #27) validated across first full epic.** Winston's design doc gave Amelia unambiguous decomposition plans for every story — zero mid-story architectural debates, zero scope surprises. Two retros of evidence: soft pre-steps fail, story-embedded gates ship.

2. **Zero-regression refactoring discipline.** 7,540 lines decomposed, 122 new tests, green suite at every intermediate step. `pnpm test` after every extraction kept the feedback loop tight.

3. **3-layer adversarial code review matured into reliable process.** Reviewer context template + triage buckets (patch/bad-spec/defer/reject) eliminated biggest noise source from 10.7. Bad-spec bucket enabled stories and code to co-evolve during review.

4. **Retro follow-through at near-100%.** Agreement #30 proven across two consecutive epics. When commitments are embedded as stories or verifiable tasks, completion rate approaches 100%.

5. **Facade pattern as reusable template.** Established in 10-8-1, replicated verbatim in 10-8-2, 10-8-3, 10-8-4 with zero controller changes. Config passthrough pattern, shared utility extraction, and spec file migration all became repeatable.

6. **Previously untested formatters caught and covered.** Lad MCP review in 10-8-5 found 6 formatters with zero unit tests — pre-existing gaps fixed during the epic.

## Challenges & Growth Areas

1. **Line count targets consistently exceeded.** Every decomposed service overshot formatted line targets due to Prettier auto-formatting expansion (~40-70% inflation from multi-argument calls, event construction, inline objects). ExecutionService: 590 logical → 1,024 formatted. ExitMonitorService: ~650 logical → 914 formatted.

2. **Constructor dependency targets universally exceeded.** Facades ranged from 6-9 deps vs. 5-dep target. Convention doesn't distinguish orchestrators from leaf services.

3. **Spec file renaming during decomposition was manual and error-prone.** Caught by code review in 10-8-2 (`exit-monitor-chunking.spec.ts` should have been `exit-execution-chunking.spec.ts`). Manual renaming of 30+ test files invites mistakes.

4. **Three consecutive internal-tooling epics without operator-facing features** (10.7 profitability, 10.8 decomposition, 10.9 backtesting). Path is clear but scope discipline in 10.9 is critical.

5. **Module provider count overages unresolved.** DashboardModule (13), ExecutionModule (15) exceed ~8 convention. Design spike noted mitigation (ConfigModule extraction) but deferred to Epic 11.

## Key Insights & Learnings

1. **Formatted line count ≠ logical complexity.** New convention: 600 formatted lines = review trigger, 400 logical lines = hard gate. Services exceeding 600 formatted but under 400 logical get documented exceptions. Both exceeded = must decompose.

2. **Facades have inherent dependency costs.** New convention: leaf services ≤5 deps, facades ≤8 deps with mandatory rationale comment. Keeps pressure on leaf services where it matters while acknowledging orchestration costs.

3. **Bad-spec triage bucket is a process maturity signal.** ACs amended in 10-8-2 and 10-8-4 based on review findings — stories and code evolve together instead of treating original ACs as immutable.

4. **Design spike → code stories sequencing eliminates scope surprises.** Repeat for every epic with structural complexity. Agreement #27 is now a proven, validated pattern.

5. **Feature work and refactoring must be sequential, not parallel.** 10.7 grew God Objects while adding features (correct). 10.8 decomposed them (correct). Interleaving creates merge conflicts and split attention.
