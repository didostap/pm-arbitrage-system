# Story 10-95.1: TimescaleDB Extension Installation & Proof of Concept

Status: done

## Story

As an operator,
I want TimescaleDB installed and verified on the smallest time-series table,
so that I can validate the migration approach before converting larger tables.

## Acceptance Criteria

1. **Given** Docker Compose files use `postgres:16` image, **When** the migration is applied, **Then** both `docker-compose.yml` and `docker-compose.dev.yml` use `timescale/timescaledb-ha:pg16` **And** existing data volumes are compatible (drop-in replacement).

2. **Given** TimescaleDB is not installed, **When** the Prisma migration runs, **Then** `CREATE EXTENSION IF NOT EXISTS timescaledb` succeeds.

3. **Given** `historical_trades` (5.9 GB, smallest table) exists as a regular table, **When** the hypertable conversion runs, **Then** `SELECT create_hypertable('historical_trades', 'timestamp', migrate_data => true, chunk_time_interval => INTERVAL '1 day')` succeeds **And** the Prisma schema uses `@@id([id, timestamp])` composite PK **And** the unique constraint includes `timestamp`: `@@unique([platform, contractId, source, externalTradeId, timestamp])`.

4. **Given** `historical_trades` is now a hypertable, **When** existing Prisma queries execute (findMany, create, createMany), **Then** all operations succeed identically to before conversion.

5. **Given** unused indexes exist on `historical_trades`, **When** the migration runs, **Then** `historical_trades_platform_contract_id_timestamp_idx` (0 scans, 676 MB) is dropped **And** `historical_trades_timestamp_idx` (0 scans, 132 MB) is dropped.

## Dev Notes

### Architecture Context

This is the first story in Epic 10.95 (TimescaleDB Migration). It's a **proof-of-concept** on the smallest table (`historical_trades`, 5.9 GB) to validate the migration approach before tackling the two massive tables in Story 10-95-2 (`historical_prices` 180 GB + `historical_depths` 151 GB).

TimescaleDB is a PostgreSQL extension, not a database replacement. The `timescale/timescaledb-ha:pg16` Docker image is a drop-in superset of `postgres:16` — same data directory format, same wire protocol, same env vars. Existing data volumes require no changes.

### Prerequisite: Database Backup

Epic 10.95 requires a full `pg_dump` backup before any migration. For this POC (5.9 GB table), the operator should back up before running the migration. Rollback from a failed `create_hypertable` requires restoring from backup — hypertable conversion is not reversible via standard migration rollback.

### Critical: Hypertable Constraint Rules

**All unique indexes and primary keys on a hypertable MUST include the partitioning column** (`timestamp` in our case). This is a PostgreSQL partitioning requirement enforced by TimescaleDB. Violations cause `TS103: cannot create a unique index without the column "timestamp" (used in partitioning)`.

This means:
- `@id` on just `id` alone is **invalid** — must become `@@id([id, timestamp])`
- `@@unique([platform, contractId, source, externalTradeId])` is **invalid** — must include `timestamp`
- The raw SQL `ON CONFLICT` in `predexon-bulk-insert.ts` must match the updated unique constraint

### Unique Constraint Semantic Note

Adding `timestamp` to the unique constraint changes deduplication: previously `(platform, contractId, source, externalTradeId)` prevented any duplicate external trade ID; now the same ID with a different timestamp would be accepted. In practice this is safe — external trade IDs from platform APIs are globally unique, and the timestamp is the trade's own timestamp (not insertion time), so re-fetching the same trade always produces the same timestamp. The ON CONFLICT DO NOTHING behavior is preserved.

### Prisma Compatibility

Prisma does not have native TimescaleDB support. The approach is:
1. Prisma schema declares composite PK and updated constraints (standard Prisma features)
2. Extension installation and hypertable conversion happen via manual SQL in the migration file
3. All standard Prisma Client operations (`findMany`, `createMany`, `aggregate`, `groupBy`, `count`) work identically on hypertables — TimescaleDB is transparent to the ORM layer
4. `time_bucket()` and other TimescaleDB functions require `$queryRaw` (not used in this story but relevant for 10-95-3)

**Do NOT use `postgresqlExtensions` preview feature** — Prisma is discontinuing it (prisma/prisma#26136). Handle extensions via manual migration SQL.
