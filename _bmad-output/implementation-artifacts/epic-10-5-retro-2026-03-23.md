## Challenges & Growth Areas

1. **Code review is the only quality gate — 5/5 critical findings caught at the expensive end of the timeline.** Every code story had critical or high-severity findings. Review is catching both *architecture* flaws and *implementation* bugs because no earlier checkpoint exists for architecture validation.

2. **Hot-reload architecture rewrite (10-5-2).** Initial `ReloadableService` interface was fundamentally broken due to NestJS module init ordering. Required complete mid-story pivot to `ModuleRef`-based lazy resolution. A 15-minute design sketch review would have caught this before implementation.

3. **Secondary reviewer unreliable — single point of failure.** The secondary Lad MCP reviewer (glm-5-turbo) consistently failed with provider errors across multiple stories. 3-layer adversarial review was effectively 2-layer for some stories.

4. **Spec-to-implementation divergence (10-5-3).** Backend used `Detection & Edge` vs. spec's `Detection & Edge Calculation`. Cascading E2E test failures for a naming mismatch. Coordination tax.

5. **Data integrity defect class has no structural guard.** Event wiring, collection lifecycle, and mode contamination all have structural guards. Financial math data integrity — functions returning incomplete or incorrect values under certain conditions — relies entirely on review vigilance.

6. **Five stories, five different failure modes.** Data integrity, architecture, concurrency, dead code wiring, mode contamination. Breadth problem (inherent to AI-generated code) rather than depth problem (fixable with training). The dev agent produces plausible code with fundamental flaws hiding underneath — this is a permanent property requiring permanent countermeasures.

---

## Key Insights & Lessons Learned

1. **"Two lighter checkpoints early > one heavy rescue late."** Design sketch gate (catches architecture flaws) + code review (catches implementation flaws) is better division of labor than code review doing both jobs. Proven by 10-5-2 where a 15-minute design review would have prevented a full architecture rewrite.

2. **Structural guards absorb recurring defect classes; review should focus on novel bugs.** After Epic 10.5, three of four recurring categories have structural prevention (wiring, collections, mode). Review energy should concentrate on data integrity and architecture — the categories without automated guards.

3. **Code review catches are a feature, not a failure.** 5/5 critical findings caught, zero production incidents. The question isn't "why is review finding so much?" but "how do we shift some of that finding earlier?"

4. **The dev agent's plausibility problem is permanent.** Five different failure modes across five stories means this isn't a single weakness to drill on. The countermeasure is layered defense (structural guards + design review + code review), not hoping for better generation.

5. **Course-corrected epics can be cohesive.** Epic 10.5 didn't exist in the original roadmap — it merged settings infrastructure with retro action items. The result was a focused, coherent epic with 100% completion. Sprint planning absorbed the course correction cleanly.

## Team Agreements

### New (Epic 10.5 Retro)

- **Agreement #27: Design sketch gate for hot-path stories.** Any story touching execution, edge calculation, or risk gets a lightweight design sketch reviewed by the architect before implementation begins. Catches architecture-class flaws before they become code review findings. Validated by 10-5-2 hot-reload rewrite that a 15-minute sketch would have prevented.

- **Agreement #28: Reviewer redundancy.** Identify a fallback reviewer before each epic starts. If the primary Lad MCP reviewer fails after 3 retries, escalate to the fallback — different model, second instance, or structured self-review protocol with documented checklist. Addresses single point of failure identified in Epic 10.5.

- **Agreement #29: Pre-implementation naming walkthrough.** Story author and implementer do a 5-minute walkthrough of string literals, enum values, DTO field names, and API path conventions before coding starts. Prevents spec-to-implementation divergence that caused cascading test failures in 10-5-3.

### Validated (Still in Effect)

- **#24: Retro commitments as deliverables.** Validated by 100% (7/7) follow-through in Epic 10.5 vs. 62% (5/8) in Epic 9→10.
- **#25: Story sizing gate.** No oversized stories in Epic 10.5 (largest was 10-5-2 at ~12 tasks, but single integration boundary).
- **#26: Structural guards over review vigilance.** Validated by 10-5-4 and 10-5-5 delivering automated guards for three recurring defect classes.
- **#22: Correction budget 30-40%.** Not tested in 10.5 (zero course corrections within epic).
- **#18: Vertical slice minimum.** Applied in 10-5-1 through 10-5-3 (backend + dashboard for settings).
