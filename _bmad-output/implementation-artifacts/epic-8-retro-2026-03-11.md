## Challenges & Learnings

### 1. Platform API Data Quality — Systemic Pattern
- **Challenge:** 12 post-implementation bug fixes in Story 8.4 alone, all from live testing against real APIs
- **Root cause:** Mocks encoded assumptions about API behavior that didn't match reality. Kalshi returns empty strings `""` instead of null, Polymarket returns JSON-encoded strings for arrays, `ConfigService.get<number>()` returns strings. The `??` vs `||` gotcha hit three stories.
- **Resolution:** Three structural defenses for Epic 9:
  1. **Real API response fixtures** — capture actual responses, test against reality not assumptions
  2. **Zod schemas at connector boundary** — runtime validation catches deviations at ingestion, not three modules downstream
  3. **Real API spike mandate** — any new integration includes hitting the real API and documenting actual response shape
- **Key insight (Arbi):** "Platform API data quality is a first-class concern, not a per-story surprise. We can't test for what we don't know, but we can build a system that fails loudly when reality deviates from our model."

### 2. Story Sizing Under Velocity Pressure
- **Challenge:** Story 8.3 (6 task groups, connectors + contract-matching + monitoring + dashboard) and 8.5 (19 tasks, 3 independent updates) bent the two-layer sizing rule. 8.3 accumulated 3 HIGH code review findings.
- **Correlation:** Story 8.3 — the story that most clearly bent the sizing rule — also had the largest test contribution (+89 tests, 43% of Epic 8's total) and the most HIGH-severity code review findings (3H). This is not coincidence: oversized stories produce more surface area for bugs, require more tests to cover, and impose higher cognitive load on reviewers. The costs were absorbed through effort, not prevented by process.
- **Root cause:** Velocity pressure — expanding story count felt like admitting underplanning (Elena). Grouping by FR requirement instead of deployability (Charlie). Grouping bugfixes as one story because they felt like "cleanup" (Alice).
- **Resolution:** Refined rule: *independently deployable with distinct failure modes = separate stories*. 15-minute sizing check at story creation when task groups > 2 or layers > 2.
- **Key insight (Arbi):** "Story count is not a cost to minimize; review cognitive load is. The real measure is whether each story has a single, clear failure mode that a reviewer can reason about in one sitting."

### 3. Scale-Sensitive Numeric Types Crossing Module Boundaries
- **Challenge:** `divergenceRate` stored as fraction (0-1) in `ResolutionContext` but percentage (0-100) in `CalibrationBand`. Both typed as plain `number`. TypeScript didn't catch it.
- **Root cause:** No structural enforcement of numeric scale semantics. JSDoc comments (vigilance) instead of branded types (constraint). Same class of bug as `confidenceScore` stored 0-100 but Epic 9.3's formula assuming 0-1.
- **Resolution:** Branded types for scale-sensitive numerics: `Percentage`, `Fraction`, `ConfidenceScore`. TypeScript compiler rejects passing wrong scale at compile time. Zero runtime cost.
- **Key insight (Arbi):** "If we fix divergenceRate with a naming convention but don't address the structural pattern, we'll hit the same class of bug with the next numeric field that crosses a module boundary."

### 4. Mock Fidelity Gap
- **Challenge:** Unit tests pass against hand-crafted mocks based on API documentation, but real APIs behave differently. 12 bugs found only through live testing.
- **Root cause:** Mocks are aspirational (what we think the API returns), not empirical (what the API actually returns).
- **Resolution:** Fixture-based testing — freeze real API responses in `connectors/*/fixtures/`, import in tests. When APIs change, tests break at the boundary.
- **Key insight (Dana):** "We can't write tests for data quality issues we don't know exist. But we can build a system where mock fidelity is guaranteed by using real data."

### 5. Silent Degradation Risk
- **Challenge:** Calibration persistence is in-memory only. Engine restart silently falls back to default thresholds, which may include the 0.30 value Story 8.6 proved kills legitimate matches.
- **Resolution:** Stopgap: serialize CalibrationResult to JSON on disk, reload on startup with staleness warning. Proper Prisma migration deferred to Epic 9.
- **Key insight (Arbi):** "Silent degradation after restart is worse than loud failure. The operator thinks the system is operating at one threshold while it's actually operating at another."

## Key Insights

1. **Platform API data quality is a systemic concern** — needs structural defense (fixtures, Zod validation, real API spikes), not per-story surprise handling
2. **Branded types for scale-sensitive numerics turn vigilance into constraints** — prevents the ÷100 class of bugs at compile time with zero runtime cost
3. **Story count is not a cost to minimize; review cognitive load is** — independently deployable with distinct failure modes = separate stories
4. **Silent degradation after restart is worse than loud failure** — no component may silently fall back to default configuration
5. **Strongest verification came from running against reality, not the test suite** — unit tests validate internal consistency; live runs validate correctness against the world. Both matter.
6. **Watched items should trigger sizing discipline, not just scope awareness** — when a planned feature triggers a watched item, treat it as a signal to split more aggressively
