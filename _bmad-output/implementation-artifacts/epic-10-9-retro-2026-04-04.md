## Successes & Strengths

1. **Design spike gate validated for third consecutive epic (Agreement #27).** Winston's design doc verified 7 external data sources before code started. Caught Kalshi's undocumented 10K candlestick truncation, Goldsky's missing side/price/blockNumber fields, PMXT's wider-than-expected file sizes, OddsPipe's lack of platform IDs. Zero mid-story architectural debates on verified sources.

2. **3-layer adversarial code review fully mature.** Standard operating procedure across 8 stories. Typically 30-70 raw findings per story, triaged to 15-25 actionable patches via triage buckets (patch/bad-spec/intent-gap/defer/reject). Bad-spec bucket continued to enable story/code co-evolution during review.

3. **777 new tests with consistent assertion depth.** Payload verification with `expect.objectContaining()`, event wiring tests via `expectEventHandled()`, dual paper/live mode matrices. These are confidence-building tests, not checkbox tests.

4. **Chunked pipeline architecture breakthrough (10-9-3a).** Transformed OOM-prone monolithic loader into scalable chunked time-window processor. Discriminated union DepthCache pattern (eager for small datasets, lazy LRU for large) is a reusable architectural pattern.

5. **Decimal-to-native-number insight (10-9-3b).** `decimal.js` Decimal objects at ~120 bytes each vs. 8 bytes for native numbers. For non-settlement calculations (VWAP estimation), native numbers are acceptable. 93% memory reduction per depth record.

6. **Kalshi dual-partition routing fix (10-9-1a-fix) unblocked 3 months of price data.** Discovered historical service only queried one partition. This single fix made production backtesting viable.

7. **Zero production incidents despite 7 external API integrations.** Every external data source (Kalshi, Polymarket, Goldsky, PMXT Archive, OddsPipe, Predexon, poly_data) integrated without production impact.

8. **Monitoring infrastructure caught storage degradation proactively.** Cache hit rates (79.5%/84.3% vs 95% target) flagged before crisis. Epic 10.95 is a planned migration, not emergency firefighting.

9. **Previous retro follow-through at 100%.** Agreement #30 (no soft commitments) validated across third consecutive epic. Story-embedded deliverables continue to ship at near-100%.

## Challenges & Growth Areas

1. **44% scope growth — 4 unplanned stories.** Planned 9, shipped 13. Near upper bound of 30-40% correction budget. Breakdown:
   - **10-9-1a-fix (Kalshi partition routing):** Structurally preventable. Design spike verified historical endpoint schema but not live-vs-historical response payload comparison. One gap in the spike checklist.
   - **10-9-3a, 10-9-3b (scalable loading + OOM fix):** Anticipated scope. Sprint status on 10-9-3 noted "production-scale loading deferred to 10-9-3a." OOM severity from `decimal.js` memory footprint was genuinely unpredictable from design doc. Correction buffer working as intended.
   - **10-9-8 (dashboard bugs):** Normal integration tax. `totalPnl` vs `netPnl` field mismatch, missing positions table. Expected, contained, done in a day.

2. **337 GB storage growth — acceptable sequence with planning gap.** Three time-series tables grew to 337 GB in ~3 months (99.9% of total). 76 GB index bloat. The build-then-optimize sequence was correct (can't optimize data you haven't ingested), but a storage risk assessment during epic planning would have flagged the trajectory and enabled parallel 10.95 planning. Key miss: JSONB depth levels at 1.4 KB/row (estimated 500 bytes) and unmodeled index amplification factor (2-3x).

3. **BacktestEngineService grew to 917 lines / 9 deps — new God Object.** Started at ~380 lines (10-9-3), reached 643 (10-9-4), landed at 917 (10-9-3a). Constructor hit 9 dependencies (above facade ≤8 threshold). Decomposition seams are visible: WalkForwardRoutingService and ChunkedDataLoadingService. Addressed by adding story 10-95-4.

4. **NestJS DI and TypeScript strict mode friction — recurring defect category.** Three distinct traps hit across 4+ stories:
   - `import type` breaks NestJS DI with unplugin-swc (10-9-4, 10-9-7)
   - `configService.get<boolean>()` returns strings, not booleans (10-9-6, 10-9-7)
   - Global prefix + controller prefix duplication — 4 controllers routed to `api/api/...` (10-9-6)
   - Meets Agreement #26 threshold (3+ occurrences across an epic). Structural guard required.

5. **Lad MCP code review intermittent failure.** Both attempts returned empty/errored on 10-9-1b (novel external integration — highest review value). Proceeded per CLAUDE.md retry-and-proceed policy. Succeeded on other stories. Monitoring note: escalate if 2+ failures in 10.95.

6. **Route prefix duplication missed by 3-layer adversarial review.** The doubled `api/api/` prefix in 4 controllers was caught only in QA, not code review. Reviewer would need whole-codebase context (global prefix in main.ts) to catch it — outside PR diff scope.

## Key Insights & Learnings

1. **The correction buffer model works — and the types of unplanned work matter.** 44% growth is informative, not alarming. One structurally preventable correction (Kalshi field divergence), two anticipated-but-harder-than-expected corrections (OOM), one integration tax. The model held.

2. **Data-heavy epics need a storage risk assessment as a first-class planning artifact.** Formula: rows/day × average row size × index amplification factor = projected footprint at 3/6/12 months. Red-line threshold triggers optimization planning. This would have enabled parallel 10.95 planning.

3. **Multi-partition APIs need response payload comparison, not just schema review.** Design spike caught 7 gotchas through API verification, but the one it missed (Kalshi field naming divergence between partitions) was the one where it only reviewed documentation instead of comparing actual responses.

4. **Don't carry decomposition debt across epic boundaries when the fix is already scoped.** BacktestEngineService decomposition has visible seams and a proven playbook (10.8 facade pattern). Slotted as 10-95-4 to break the grow-then-decompose cycle rather than deferring to Epic 11.

5. **Profitability validation produced a strategic decision, not the hoped-for outcome.** 0% profitability across 202 paper positions led directly to Epic 10.9's existence. The system's execution mechanics work; parameter calibration had no empirical foundation. Backtesting and calibration now provide the tools to answer the profitability question with data.

### New Conventions from 10.9

- **Multi-partition API verification:** Design spikes for APIs with multiple partitions or endpoint variants must include actual response payload side-by-side comparison, not just schema documentation review.
- **Storage risk assessment for data-heavy epics:** One-page estimate (rows/day × row size × index factor = projected footprint at 3/6/12 months) with red-line threshold. Required for any epic introducing new data ingestion or significant write volume.
- **Correction buffer upper bound is informative:** 44% against 30-40% budget validates the model. Track unplanned story types to distinguish structurally preventable corrections from acceptable emergent work.

## Final Assessment

Epic 10.9 delivered 13 stories across 8 days, building the complete backtesting and calibration capability: 7-source data ingestion, parameterized simulation engine, sensitivity analysis with walk-forward validation, and a full dashboard page. 777 new tests, zero production incidents, and 100% follow-through on Epic 10.8's action items.

The team is well-positioned for Epic 10.95 with clear preparation requirements, a 4-story scope (3 migration + 1 decomposition), and two structural guards landing before migration work begins.
